# CLAUDE.md

## 0. このファイルの位置づけ

本ファイルは、本プロジェクト(tokyo-nova-axleration / TNX)の開発運用における**唯一の正本**である。Claude Code は作業着手時にまず本ファイルを読むこと。

本ファイルは**規範と指針**に徹する。プロジェクトの知識・記録の実体(設計判断の経緯、ルール解釈、既知不具合の詳細、フェーズ記録、アーキテクチャ詳細、ロードマップ等)は、別リポジトリ `llm-wiki/` に集約されている。本ファイルはそれらを**直接複製せず**、必要な箇所へ参照で誘導する(§5 参照プロトコル)。

矛盾が生じた場合の優先順位は、**実コード > 本ファイル > llm-wiki** とする。本ファイルが古い記述を含む場合は、実コードを正とし、本ファイルの該当箇所の更新を提案すること。

## 1. プロジェクト概要

TNX は、TRPG「トーキョーN◎VA THE AXLERATION」(5th edition、FEAR 系)を遊ぶための Foundry VTT カスタム ゲームシステムである。最終的に Foundry の公式パッケージ登録での公開を志向し、MVP ではなく実走志向(各機能を最低限でなく作り込む)で開発する。

ルール上の前提(設計判断の背景として保持):
- 判定はカード(Cards API)ベースで行う。ダイスロールを用いない。
- 1 アクトはシーンの連なり(シーン制)で進行する。
- 神業はゲーム進行を書き換えるブレイクスルーであり、通常の数値計算では表現できない。
- 経験点はキャラクターではなくプレイヤー(=User)に付与される(FEAR 系の特徴)。

ライセンスと著作権:
- 本システムのコードは MIT ライセンスで配布する。
- Foundry VTT の Limited License Agreement for Package Development に従う。
- 「トーキョーN◎VA THE AXLERATION」は「鈴吹太郎」「有限会社ファー・イースト・アミューズメント・リサーチ」および「株式会社KADOKAWA」の著作物。本プロジェクトはその二次創作物である。

開発規模の見通し:長期戦(年単位)を前提に、技術的負債を避けて堅く作る。

ロードマップ・現在のフェーズ・前フェーズの完了内容は llm-wiki の Roadmap / Phases に置く(§5)。**現在のフェーズと次の作業対象は、作業着手時に必ず wiki の現フェーズページで確認すること。**

## 2. 開発体制とワークフロー

### 2.1 役割分担

- **ユーザー**: TNX のルール知識の言語化、設計・実装方針の最終判断、Foundry 実機での動作確認、git のコミット承認・タグ付け・push、ルールの一次情報はユーザーが言葉で提供する。
- **Claude Code**: コード調査・設計・実装・テスト作成・記録(監査/設計レビュー/既知不具合)の一貫した担当。ルールとコードの照合は Code 側が行う。

ルール解釈そのものはユーザーが権威を持つ。Code はそれを前提に設計・実装する(§4)。

### 2.2 作業の進め方

- 各フェーズはサブフェーズに細分し、着実に仕上げる。「最小実装」という発想は採らず、各機能を作り込む。
- **承認は設計の確定時にまとめて取る。** 大きな変更や複雑なシステム(判定・神業など)は「設計フェーズ → 実装フェーズ」に分け、設計フェーズでルールの言語化・データ構造(追加・改名するフィールドを含む)を確定してユーザーの承認を得る。**承認済みの設計に沿うサブステップは、逐一の再承認を要しない**——設計から外れる判断が生じたときに確認する。
- 1 つの作業単位に「複数の無関係な不具合修正を混ぜない」(論理単位でコミットを分ける)。
- 新規機能の実装には、可能な範囲でテストを併設する。

### 2.3 二リポジトリ運用

本プロジェクトは二つの git リポジトリにまたがる。

- **本体リポジトリ**(`tokyo-nova-axleration/`): システムのコード、本ファイル(CLAUDE.md)。
- **llm-wiki リポジトリ**(`tokyo-nova-axleration/llm-wiki/`): 知識・記録の正本。本体から `.gitignore` で除外される。パス別リポジトリ、Obsidian で管理。

運用規範:
- コードの変更は本体に、記録(設計レビュー・メカニクス監査・既知不具合)の追記は llm-wiki にコミットする(§6)。
- 1 つの作業がコード変更と記録更新の両方を伴う場合、**両リポジトリそれぞれにコミットする**。本体のコード変更コミットと、それに対応する wiki の記録更新コミットを、同じ作業セッション内で漏れなく両方仕上げること(本体だけ進めて記録が遅れた状態を残さない)。
- どちらのリポジトリも push はユーザーが手動で行う。Code はコミットまで。
- llm-wiki で作業するときは、カレントが llm-wiki リポジトリであること(`git rev-parse --show-toplevel` が llm-wiki を指すこと)を確認してから操作する。

### 2.4 開発コマンド

- `npm install` — 依存導入
- `npm test` — テスト全実行
- `npm run test:watch` — テスト監視実行
- `npm run lint` — リント
- `npm run lint:fix` — リント自動修正
- Foundry の起動・リロードはユーザーが手動で行う。

### 2.5 長期離脱後の再開

- **2 週間以上**空いた場合: 本ファイルと、wiki の現フェーズページを見直してから着手する。
- **3 ヶ月以上**空いた場合: wiki から現状を再構築して着手する。

## 3. アーキテクチャの要点

詳細な構造説明は llm-wiki の Architecture に置く(§5)。本節は Code が踏み外さないための要点に絞る。

- **DataModel(TypeDataModel)ベース。** `template.json` は廃止済み。**復活させないこと。** ドキュメント型の定義は `documentTypes`(system.json)と `scripts/data/` 集中の DataModel が権威。
- **型の一覧・型数を本ファイルに直書きしない。** Actor / Item / Card の各 type 数・一覧は `scripts/data/` と system.json が正とする。フェーズ進行で増減するため、本ファイルに数値を書くと陳腐化する。
- Actor types は player が廃止済み(cast / guest / troop / extra)。
- **User は DataModel を持てない。** EXP・履歴・手札関連(手札上限を含む)は User flag に保持される。手札上限の権威は User にある。実効上限の計算はゲーム設定 `defaultHandMaxSize` との連携を経て解決される。flag の読み書きは専用ヘルパー(`scripts/module/user-flag-schema.mjs`)を必ず経由すること。
- 主要モジュール: カード操作ハンドラ(`TnxActionHandler`)、常駐 HUD(`TnxHud`)、アクト用 JournalSheet(`TnxScenarioSheet`)、ready フックでの EXP 再計算。
- 派生値計算は `prepareDerivedData`(フェーズ9 で確立)で行い、実効値(能力値・制御値・アイテム実効値)はここに一本化する。ActiveEffect はこの計算経路に流れ込む。詳細は wiki の Game_Mechanics / Active_Effects と Architecture / DataModel。
- アイテムのロール処理は D&D 5e の思想を踏襲し、usage ドキュメントとして管理される。usage テンプレート(actions[])と UsageCreationDialog は判定システムの基盤として設計済み(改変は §4 で制限)。
- 命名規則: ファイル名は kebab-case、クラスは PascalCase、パッケージ ID・フラグスコープは system.json を踏襲する。

## 4. 不可侵の規範

以下は環境・フェーズにかかわらず常に守る。**不可逆・対外的な操作、またはルール解釈に関わる判断**は、着手前に確認する。それ以外(可逆でルール解釈を伴わない作業)は、必要なら**前提を明示したうえで進め**、確認のために手を止めない。

### 4.1 著作権遵守

- ルールブックのテキスト、ゲームデータをコードやドキュメントに**無断複製しない**。効果テキストの一字転記、特技・神業等の網羅的なリスト化を行わない。**公開外向けの llm-wiki であっても同様。**
  - ※ここで禁じるのは**原文の逐語転記・創作的表現の複製**であり、ユーザーが言い換えたルール内容・メカニクスを**開発用 llm-wiki に構造的に記録すること**は含まない(判定・ダメージ・神業などすべて同基準で記録する)。著作権配慮は**配布物本体(コード・compendium 同梱データ)への焼き込み回避**で行う(効果文は Item へのユーザー投入・入力機能で対応)。
- 判定ロジック等の実装に必要な範囲の「enum・定数といった**構造的最小限**」の値はコードに含めてよい。
- ルールの**構造・仕組み**(データフロー、状態遷移、判定の関係)の記述、および**ユーザーが引き出して記述したルール解釈**に基づく実装・設計・記録は可。ルールの一次情報はユーザーが書く。Code はそれを前提とする。

### 4.2 ルール解釈の確定

- ゲームルールの解釈に関わる判断を**推測で実装しない**。判定の達成値計算・神業の挙動・セッション進行など、解釈が要る箇所はユーザーに確認する。
- 神業の自動化は計算で導けないため、**設計合意のうえで実装する**(usageCount の保守など既存の機械的挙動は可)。

### 4.3 変更の抑制

- **動いている既存システムを、理解と合意なしに書き換えない。** 「改善のついで」「効率化のため」でのリファクタ・大規模移行・全体書き換えを始めない。必要なら提案にとどめ、設計フェーズで合意してから着手する。例: EXP 同期機構は一見冗長でも意図的な設計(権威は User flag・ドキュメント跨ぎの同期・競合リスク KI-008)であり、独断の書き換えは整合を壊す(理由は wiki の Game_Mechanics / Experience_Points)。
- **先回りの拡張をしない**(YAGNI)。「将来のために」拡張ポイントを作らず、必要になった時に追加する。現在のフェーズに該当しない作業は着手しない(必要な気づきの共有・提案は可)。
- usage テンプレート・`usage-list.hbs`・UsageCreationDialog は判定システムの基盤のため、設計議論を経ずに構造を改変しない。
- DataModel のフィールド追加・改名は、設計フェーズでの一括承認に含める(§2.2)。

## 5. llm-wiki 参照プロトコル

llm-wiki は別リポジトリのため、Code の起動時自動読み込みには乗らない。したがって、**該当領域の作業に着手する前に、対応する wiki ページを明示的に読むこと。** 本ファイルの規範と、wiki の知識を併せて初めて作業の前提が整う。

入口は wiki のインデックス(`llm-wiki/` の Index / Documentation_Map)とする。領域別の目安は以下(再編後のパス確定は §6 / 第一段階に従う)。

- 既知不具合・KI 一覧 → Issues(Active / Resolved)
- 設計判断の経緯・メカニクス監査 → System の Audit_and_Review
- ゲーム機構(EXP・カード操作・判定など) → Game_Mechanics
- フェーズ記録・現在地・ロードマップ → Phases / Roadmap
- アーキテクチャ詳細・DataModel・ドキュメント型 → Architecture / Documents
- リリース手順 → System の Release_Process

設計・実装に入る前に、扱う領域が上記のどこに当たるかを見極め、該当ページを読む。軽微・自明な作業まで儀式化はしない——根拠が要る設計・実装で読む。

### 5.1 正本に当てて検証する

実装・修正は、記憶や要約でなく**正本(wiki / コード)の原文に当てて検証してから**行う。仮説は自由に立ててよいが、**動かす前に根拠となる該当行を引用し、文の主語・条件・挙動が一致するかを逐条で確かめる**(数ターン前の自分の言語化も、読み直さずに使うと一般形へ滑るので、原文に取り直す)。

- メカニクスは「入力欄の集合」でなく**挙動そのもの**で符号化する(値が「どう決まり・何をするか」を落とさない。例: 衰弱の対象・数字は『引いて決まる』のであって選択ではない)。
- **自分のモデルから書いたテストで自分を検算しない**。同根のテストは誤りを通すので、正本と突き合わせて検証する。

## 6. 記録運用

記録の正本はすべて llm-wiki にある。本体リポジトリには記録系ファイルを置かない(`docs/` は廃止済み)。

### 6.1 記録の種別と書き込み先

- **メカニクス監査**(ルール解釈の正しさの記録)と**設計レビュー**(技術的妥当性の記録)は、wiki の System / Audit_and_Review に追記する。**この二つは関心が異なるため、混同しないこと**(ルール解釈の妥当性と、実装の技術的妥当性)。
- **既知不具合(KI)**は、wiki の Issues が**唯一の正本**である。未解消は Active、解消・分割済みは Resolved に置く。

### 6.2 記録のタイミング

- 新規実装時: メカニクス監査・設計レビューに各 1 エントリを追加する。
- 既存実装に影響する変更時: 該当エントリのステータスを更新する。
- フェーズ完了時: 全エントリを通読してステータスを最新化する。
- 記録の書き手は Code、最終承認はユーザー。

### 6.3 KI 管理

- 新たな不具合が発覚したら、wiki の Issues の Active に起票する。
- 解消・分割した KI は、Active から Resolved へ移動する。
- KI の正本は wiki の Issues 一箇所であり、他の場所に重複管理しない(従来の複数箇所管理は廃止)。

## 7. バージョニング / リリース

- SemVer に従う。現在は 0.x.x(開発中)。
- フェーズ完了ごとに MINOR を上げる。フェーズ間のバグ修正で PATCH を上げる。
- 最終フェーズ完了時に 1.0.0 をリリースする。
- Conventional Commits に準拠し、`<type>: <description>` 形式。主なタイプは feat / fix / docs / chore / refactor / test / style。**description は日本語で簡潔に。**
- バージョン番号は package.json / system.json / git タグの 3 箇所で**手動同期**する。
- リリースの詳細手順は wiki の System / Release_Process を参照する。
- タグ付け・push・GitHub Releases の作成はユーザーが手動で行う。
